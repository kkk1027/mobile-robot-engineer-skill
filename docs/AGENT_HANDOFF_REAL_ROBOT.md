# 实机轮式机器人操作：智能体交接文档

本文件提供给接手的智能体。目标是将已有轮式机器人软件安全地完成诊断、二次开发、部署和受控验证；它不授予任何未由用户明确给出的权限。

## 0. 先读与工作范围

开始前必须阅读：

1. [主 Skill](../.agents/skills/engineer-wheeled-robot-systems/SKILL.md)。
2. [SSH 实机操作与安全](../.agents/skills/engineer-wheeled-robot-systems/references/ssh-real-robot-safety.md)。
3. 若任务为二次开发或功能集成，再读 [二次开发、功能集成与快速验证](../.agents/skills/engineer-wheeled-robot-systems/references/secondary-development-and-integration.md)。

默认目标是：先在 G0/G1 完成静态、仿真或台架验证；获授权后才进入 G2 只读实机诊断；G3 与 G4 按次确认。不得把“用户允许 SSH”解释为可部署或可驱动机器人移动。

## 1. 向用户收集的最小信息

在操作前确认以下项目；会改变安全或接口结论而又未知的信息必须追问。

| 项目 | 需要确认的内容 |
| --- | --- |
| 机器人身份 | 型号/编号、底盘运动学、当前物理位置、是否架空、当前任务状态 |
| 操作等级 | 本次仅 G0/G1、允许 G2、允许具体 G3 项，或允许具体的 G4 短时动作 |
| 现场安全 | 急停位置与可达性、人员位置、场地边界、速度/加速度/时长上限、watchdog 行为 |
| 连接方式 | 用户预先配置的 OpenSSH alias；首次连接由用户核验主机指纹 |
| 软件现场 | ROS 发行版、工作区、RMW、`ROS_DOMAIN_ID`、命名空间、启动方式、运行版本/提交 |
| 变更边界 | 可读取目录、可修改目录、部署目标、维护窗口、回滚版本和负责人 |
| 验收目标 | 要验证的客户需求、量化指标、停止条件和必须留存的证据 |

不得索取或输出密码、私钥、访问 Token。只使用 `ssh <alias>` 与已配置的 ssh-agent/密钥链；禁止 `StrictHostKeyChecking=no`，禁止擅自删除 `known_hosts`。

## 2. 授权闸门

| 阶段 | 允许操作 | 开始条件 |
| --- | --- | --- |
| G0 | 阅读文档/源码、检查配置、离线日志和静态校验 | 默认允许 |
| G1 | 本地构建、仿真、单测、集成测试、fake transport | 默认允许，说明 CPU/GPU/磁盘影响 |
| G2 | SSH 只读身份检查、日志、ROS 图、诊断、频率与 TF 查询 | 用户明确授权本次连接和目标 alias |
| G3 | 部署、写参数、重启服务、控制器切换、固件/容器变更 | 每项操作前确认精确命令、目标、影响、备份与回滚 |
| G4 | 解锁、使能、发布运动命令、转向、任何执行器动作 | 每次明确授权当前动作，并完成第 6 节全部检查 |

只要出现身份不符、急停不可达、现场人员未确认、反馈异常、版本不明、无回滚、命令所有权不唯一或权限描述含糊，立即停止在当前等级并向用户报告。

## 3. 推荐工作流

### A. G0/G1：先建立可验证的方案

1. 冻结用户提供的硬件、技术栈、任务和权限事实；未知项标为 `question` 或 `blocked`。
2. 对每个已有 ROS 工作区做静态盘点，生成 `inventory.json` 与候选 `module_manifest.json`；补齐接口、修改策略、成熟度和复用结论。
3. 建立 `requirements_trace.json`、`integration_contract.json` 和（需要运动闭环时）`runtime_graph.json`。
4. 先构建最小垂直切片：一个需求、必要传感/状态、唯一命令仲裁、安全停止、执行器反馈和一个验收测试。
5. 先运行单测、静态契约检查、仿真与故障注入；把每一步的命令、版本、退出码、日志位置和结论写入工作记录。

默认将用户/厂商模块固定为只读；在适配层、能力层与 bringup 层新增代码。若必须修改原模块，先得到用户对目录与变更范围的授权，并提出回归方案。

### B. G2：SSH 只读诊断

在用户授权连接后，先确认 alias 对应的机器人身份与现场状态。只读信息收集可按以下顺序进行（根据现场 ROS 环境调整，勿猜测工作区路径）：

```bash
ssh <robot-alias> 'hostname; date -Is; uname -a; uptime; df -h'
ssh <robot-alias> '<用户确认的 ROS 环境命令> && ros2 doctor --report'
ssh <robot-alias> '<用户确认的 ROS 环境命令> && ros2 node list && ros2 topic list -t'
ssh <robot-alias> '<用户确认的 ROS 环境命令> && ros2 lifecycle nodes && ros2 action list -t'
```

按故障假设补充有限时的检查：`ros2 topic info --verbose <topic>`、`ros2 topic hz <topic>`、`ros2 topic delay <topic>`、`ros2 param dump <node>`、`ros2 run tf2_tools view_frames`、`ros2 bag info <bag>`。高带宽图像/点云不得无限 `echo`；对持续命令设置超时和输出上限。

收集结束后，在本地给出“现象 → 证据 → 假设 → 最低风险证伪检查 → 建议变更 → 回滚”的计划，再请求 G3 或 G4 授权（如需）。

### C. G3：部署或配置变更

每次变更前必须向用户清楚说明：精确目标主机与目录、将执行的命令、预计影响、备份位置、停止条件和回滚命令。确认机器人静止、任务已取消、运动授权已撤销后：

1. 记录当前代码提交、容器/服务版本、配置校验和和运行状态。
2. 备份明确的配置文件；不要批量覆盖未知路径，不要清理整个工作区，不要在远端直接拉取未经审查的代码。
3. 先做 dry-run、构建或仿真；以原子替换方式部署已验证产物。
4. 若重启服务，逐项验证 heartbeat、diagnostics、控制器状态、ROS 图、参数和超时停机。
5. 若结果不符或出现异常，立即执行已确认的回滚，保留日志并报告。

### D. G4：受控实机运动

仅在用户明确授权当前动作后执行。每次动作前同时满足：

- 机器人身份、朝向、架空/落地状态和场地边界已确认；现场人员已知晓测试并处于安全位置。
- 物理急停可达、已知其作用范围；速度、角速度、加速度与测试时长均有保守上限。
- 命令停止、SSH 断开或 DDS 断开后，底层 watchdog 能在预期超时内将执行器置零。
- 只有一个命令权威者可到达底盘；无 TF 双写、无多发布者竞态、无未处理故障。
- 已先验证零命令和超时停止；准备了独立于持续发布的停止机制。

严格按“单轮/架空 → 低速短脉冲直线 → 低速转向 → 客户场景”的顺序升级。不要使用无限循环持续发布 `cmd_vel`。一旦出现反馈符号错误、姿态异常、异响、电流异常、轨迹偏离、诊断报警、人员进入边界或网络不稳定，立即停止动作、撤销使能并报告。

## 4. 二次开发与集成检查表

交付任何“功能已集成”的结论前，确认：

- 模块清单记录了来源、提交/版本、许可证、可修改范围、接口、成熟度、验证证据与复用分类。
- 每项 P0/P1 客户需求在 `requirements_trace.json` 中有模块、接口和集成级证据；单元测试不能单独证明完成。
- `integration_contract.json` 中每个连接都是 `produces → consumes`；类型不一致有显式 adapter；动态 TF 只有唯一 owner。
- 所有运动来源都经过唯一命令权威者；权威者输出可到达执行器，且具备超时/急停停止路径。
- 新驱动有参数合同、连接/诊断状态、fake/台架测试和安全默认值，未直接接入整机运动链。
- 结论与证据匹配：未验证内容标为 `planned`、`partial` 或 `blocked`，不得宣称实机已验收。

## 5. 禁止事项

- 不绕过物理急停、安全控制器、制动、限位、watchdog、权限或硬件保护。
- 不以 SSH、源码读取权限或一次历史授权替代本次 G3/G4 确认。
- 不在未知实机上运行破坏性清理、递归删除、全盘覆盖、未审查的 `git pull`、未知固件刷写或批量参数写入。
- 不读取/打印/复制凭据，不在命令行、日志或结果中泄露敏感数据。
- 不将“包存在”“编译通过”“仿真运行”或“代码相似”表述为“客户需求已完成”或“实机可安全动作”。
- 不为掩盖问题而增加无限 timeout、多个 `cmd_vel` 发布者、临时 TF 重发或未记录的 remap。

## 6. 每次 G4 前后的最小记录

动作前记录：机器人身份、现场人员、急停状态、架空/落地、软硬件版本、起始位姿、速度/加速度/时长上限、命令权威者、停止机制、授权原文及时间。

动作后记录：实际动作、起止时间、退出码、最大速度/电流/异常、diagnostics、执行器反馈、是否按预期停止、日志/rosbag 路径、结论及下一步。只保存去敏后的必要信息。

建议以如下结构向用户交接：

```text
阶段与授权：G2 / G3 / G4，授权时间与范围
目标与验收：<需求和指标>
事实与版本：<机器人身份、提交、配置校验和>
执行与证据：<命令摘要、日志、退出码、观测>
结果：passed / partial / blocked
风险与停止条件：<已触发或仍存在的风险>
回滚状态：<未变更 / 已验证回滚点 / 已回滚>
下一步：<最低风险的下一项检查或需用户决定的事项>
```

## 7. 可直接给接手智能体的任务模板

```text
阅读 docs/AGENT_HANDOFF_REAL_ROBOT.md 和 engineer-wheeled-robot-systems Skill。
目标：<客户需求>。
已知硬件/软件：<底盘、传感器、ROS、计算平台、已有源码>。
本次授权：仅 G0/G1；或允许通过 SSH alias <alias> 进行 G2 只读诊断。
禁止事项：<例如不部署、不重启、不发布运动命令>。
验收：<量化指标或待收集的证据>。
如需 G3/G4，先给出精确计划、影响、回滚和停止条件，等待我的逐项确认。
```

只有当用户明确把本次具体动作提升到 G3 或 G4，且满足相应前置检查后，才可继续下一等级。

## 8. 从零搭建仿真机器人项目：上手流程

本节适用于用户希望从零创建轮式移动机器人仿真项目的场景。默认只使用 G0/G1：文件、构建、仿真和测试可以进行；不得把仿真结果直接推广到实机，也不得尝试连接或控制真实设备。

### 8.1 首轮输入与目标冻结

先收集下表中已知内容，并为每项记录来源（用户、数据手册、已有源码或测试假设）。不会影响仿真闭环的缺失项可使用显式的 `test_default`；会影响运动学、传感器接口、坐标、时间或安全链的缺失项必须询问。

| 输入 | 最小内容 |
| --- | --- |
| 项目目标 | 项目名、目标场景、客户需求、验收指标、仿真范围 |
| 底盘 | 差速/阿克曼/全向/舵轮、轮数、轮径、轮距/轴距、速度与加速度上限 |
| 设备 | 编码器、IMU、雷达/相机/GNSS 等传感器，已知频率、安装位姿、噪声和视场 |
| 软件 | ROS 2 发行版、Ubuntu、C++/Python、目标仿真器（Gazebo/Isaac Sim/MuJoCo 等） |
| 系统 | 单机/多机、命名空间、RMW、是否需 ROS 1 bridge、导航/建图/跟随等能力 |
| 约束 | 可用 GPU/CPU、许可证、已有依赖、允许使用的开源包与禁止项 |

把输入写成结构化 `project_intake.json`，并运行 `validate_project_intake.py`。声明预期交付等级：没有运行证据时为 L1；能构建并实际运行的仿真闭环才可称为 L2。

### 8.2 先设计，后创建包

在生成代码前完成以下四项合同，并把每一项事实同步到后续产物：

1. **机器人合同**：运动学、尺寸/质量惯量假设、轮子与关节命名、执行器和传感器、坐标系、频率、单位、速度上限与命令超时。
2. **接口合同**：Topic/Service/Action、消息类型、发布者/消费者、QoS、时间源、命名空间与 lifecycle 责任；每条动态 TF 只有一个 owner。
3. **控制合同**：唯一命令仲裁者，且 `cmd` 请求能经安全门抵达控制器；停止/超时后输出归零；里程计和传感器反馈能抵达状态估计。
4. **验收合同**：每个客户需求至少指定一个可自动或可复现的仿真场景、观测量、阈值和证据路径。

写 `robot_contract.json`，需要完整运行闭环时再写 `runtime_graph.json`，并使用 `validate_robot_contract.py`、`validate_runtime_graph.py` 检查。不得等到 launch 失败后才补 TF、单位或命令所有权。

### 8.3 建议的 ROS 2 分层

在用户指定项目目录内采用按职责划分的包，避免把 URDF、控制、算法与启动文件混在一个包中：

```text
<workspace>/src/
  <robot>_description/          # URDF/Xacro、网格、静态 TF
  <robot>_hardware/             # 仿真硬件/ros2_control 配置、执行器语义
  <robot>_control/              # 底盘控制器、里程计、命令限幅
  <robot>_sensing/              # 传感器插件与数据规范化
  <robot>_state_estimation/     # 融合与定位状态
  <robot>_navigation_decision/  # 导航、任务决策、命令请求
  <robot>_monitoring_safety/    # watchdog、仲裁、diagnostics
  <robot>_bringup/              # 组合启动、参数 profile
  <robot>_simulation_tests/     # 场景、launch 测试、故障注入
```

包名、frame、Topic 名和参数要有统一前缀规则。不要让仿真插件、导航器和遥控器同时直接发布到执行器命令；它们只能向仲裁者提交请求。

### 8.4 最小可运行垂直切片

先做能够证明闭环的最小场景，而不是一次集成建图、导航、视觉与全部传感器：

1. 启动仿真世界、`robot_description` 和 `/clock`；加载模型后检查关节、碰撞和质量惯量合理性。
2. 启动底盘控制与里程计；检查 `odom → base_link` 的唯一 TF、轮速反馈和单位。
3. 接入一个必要传感器（例如 2D 雷达）及状态估计；检查时间戳、frame、频率、噪声与传感器可见性。
4. 接入唯一命令仲裁/安全节点；用短时、受限的仿真命令验证请求 → 仲裁 → 控制器 → 运动 → 里程计反馈路径。
5. 注入命令超时、传感器掉线、错误 TF 或 QoS 不匹配；验证命令归零、诊断可见、需求状态不会被错误标为完成。
6. 仅在该切片有证据后，再加入定位、导航、建图或客户功能；每增加一项能力都重复相同的契约与回归检查。

### 8.5 L2 仿真验收清单

可声明 L2 前，至少保留以下可复现证据：

- 构建命令、依赖版本、启动命令、ROS 发行版、仿真器版本和退出码。
- 启动时的 ROS 图、TF 图、控制器状态、关键 Topic 类型/频率和 diagnostics。
- 一个受控运动请求到执行器、里程计/状态估计反馈、安全门和停止超时的闭环记录。
- 至少一个异常注入结果（如命令超时或传感器掉线）及预期安全行为。
- `generation_trace.json`、`robot_contract.json`、`runtime_graph.json` 与需求追踪的校验结果。

如果模型能显示但 TF 断裂、命令没有唯一仲裁者、执行器无反馈、时钟混乱，或测试没有证据，则只能报告 L1/`partial`，不能报告 L2。

### 8.6 可直接交给接手智能体的从零仿真任务模板

```text
阅读 docs/AGENT_HANDOFF_REAL_ROBOT.md 和 engineer-wheeled-robot-systems Skill。
在 <项目目录> 从零创建 <项目名> 的 ROS 2 <发行版> 仿真工程。
底盘：<运动学、轮数、轮径、轮距/轴距、速度上限>。
传感器：<设备、frame、频率、已知安装位姿>。
平台与仿真器：<Ubuntu、CPU/GPU、Gazebo/Isaac Sim/MuJoCo>。
目标场景与验收：<例如室内导航、1 m/s 以下、命令超时后停止>。
本次仅授权 G0/G1，不连接、不部署、不控制真实机器人。
先输出 project_intake、机器人/接口/控制/验收合同和计划；缺失但影响结论的信息先提问。
随后实现最小垂直切片，运行静态校验、构建和仿真测试。
最终交付 L1/L2 级别、启动/测试命令、证据路径、未验证项、风险与下一步；不要把仿真结论写成实机验证。
```

该模板的目标是让接手智能体先建立可重复、可审计的仿真闭环。未来要迁移至实机时，须新建 L3/L4 计划，补齐真实硬件合同、部署/回滚方案和 G3/G4 的逐项授权，不能沿用本节的默认权限。
