# 决策、导航与控制

## 任务层

使用行为树、状态机或任务调度器表达任务，不把业务逻辑塞入控制回调。每个行为定义：

- 前置条件、成功、失败、取消和超时。
- 资源所有权和运动命令租约。
- 恢复次数、回退路径和人工接管条件。
- 可观测状态和事件记录。

Nav2 行为树适合导航任务编排；复杂业务可在其外层使用任务状态机，但保持单一运动命令仲裁。

## 全局规划

根据地图、底盘约束和任务选择：

- 差速/全向低速：网格搜索或采样规划，明确是否允许原地旋转和横移。
- 阿克曼：使用满足曲率和最小转弯半径的状态空间，避免不可执行的折线。
- 舵轮：根据当前模式和模块约束决定是否按全向或非完整模型规划。

规划器输出必须由控制器可跟踪；地图分辨率、footprint、曲率和速度约束一致。

## 局部规划与控制

选择 DWB、MPPI、TEB 类方法、Pure Pursuit、Stanley、MPC 或自定义控制器时，先写明：

- 控制输入和状态。
- 运动模型。
- 预测/采样周期。
- 约束和代价。
- 计算预算和最坏延迟。
- 失败时的安全行为。

阿克曼不能使用要求原地旋转的恢复动作；麦克纳姆/全向轮要确保规划器和控制器都启用横向自由度。

## ros2_control

当上位机需要统一管理关节接口、控制器切换和硬件插件时，优先通过 hardware_interface 暴露 joint state/command interface。若经验证的 MCU 固件直接订阅底盘命令并发布里程计，或仿真器原生插件提供同一上层契约，可保留该实现；不要为了形式一致再叠加一个重复控制器。

选择 ros2_control 时检查：

- read、update、write 周期和 executor 关系。
- controller_manager update_rate。
- 控制器 claimed interfaces 和生命周期。
- joint limits、软限位、速度/加速度限制。
- 控制器切换时的命令连续性。
- 硬件错误返回与 recovery 行为。

无论采用哪种实现，都不要让多个节点直接写电机总线或同时拥有同一底盘命令。

## 命令整形与安全

在最终执行前应用：

- 速度、角速度、加速度、减速度和 jerk 限制。
- 底盘运动学可行域。
- footprint/制动距离相关的动态限速。
- 命令陈旧超时和零速保持。
- 急停、保护停和任务取消的优先级。

区分“导航无目标”“控制器无输出”和“安全层主动置零”，并发布原因码。

## 控制验证

由低风险到高风险：

1. 纯数学单元测试和限制器测试。
2. 仿真阶跃、斜坡、圆轨迹、八字和路径跟踪。
3. 延迟、噪声、丢包、摩擦和载荷扰动。
4. HIL 和架空轮。
5. 低速受控实机。
6. 任务速度和复杂环境。

记录横向误差、航向误差、稳态误差、超调、停止距离、控制饱和、周期抖动和能耗。

## 恢复与降级

- 规划失败：区分地图、约束、起终点和资源问题。
- 控制失败：停止后再恢复，不在未知状态继续输出。
- 定位失效：限速、原地等待或安全停止，避免盲目旋转。
- 计算过载：优先保证安全链和底盘 watchdog，降低非关键感知频率。
- 通信断开：底层在独立超时内停车。
