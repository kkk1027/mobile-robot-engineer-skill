# 系统信息收集与路由

## 输入容器

用户只提供“硬件设备、技术栈、项目方向”是合理的，但这三项是信息容器，不是固定的三个字符串。为了构建完整项目，容器内部必须能回答会改变架构、安全或实机接口的问题。另用 `execution_boundary` 记录访问权限、部署目标和实机运动授权。

建议把输入保存为 JSON，并为来源不直观的值附上 `provenance`：

- `user_confirmed`：用户明确确认。
- `source_confirmed`：从获准读取的源码、配置或硬件文档确认。
- `measured`：通过可复现测量确认，并记录测量方法。
- `test_default_simulation_only`：仅用于仿真测试。
- `test_default_bench_only`：仅用于断开动力或受控台架测试。
- `unresolved`：尚未确认。
- `conflict`：来源之间矛盾，必须询问或测量。

任何 `test_default*`、`unresolved*` 或 `conflict*` 值都不能解锁实机运动。不得把测试默认值静默改写成真实硬件事实。

## 主动补全流程

1. 固化用户首轮输入，区分事实、假设和目标。
2. 若有获准读取的仓库，先运行 `inspect_ros_workspace.py`，只提取输入范围允许的硬件、技术栈和项目方向；不要要求用户重复源码中可发现的信息。
3. 运行 `validate_project_intake.py`，生成交付等级、缺口和问题。
4. 只询问会改变当前目标等级的问题。一次按“运动与驱动、安全与供电、传感器与标定、软件与验收”的顺序给出成组问题。
5. 用户允许测试默认值时，逐项标注来源和适用范围；不能用默认值推断电机引脚语义、驱动真值表、制动逻辑或电气安全链。
6. 补全后再次验证，冻结输入文件及其 SHA-256，再开始独立生成。

## 构建信息清单

- 任务：建图、定位、导航、巡检、搬运、跟随、遥控、编队或研究目标；给出任务优先级和验收指标。
- 底盘：轮型、轮数、驱动轮/转向轮、轮径、轮距/轴距、最大速度、负载、是否能原地转向。
- 驱动：电机与驱动器型号、额定/峰值电压电流、编码器分辨率与减速比、控制模式和闭环所在计算节点。
- 驱动接口：PWM+DIR、双输入 H 桥、CAN/CAN-FD、EtherCAT 等；必须确认 enable、brake、fault、方向极性和真值表，不能由“有两个引脚”推断。
- 传感器：型号、接口、频率、量程、坐标外参、时间戳来源、噪声和标定状态。
- 计算：MCU、Jetson、工控机的型号、操作系统、CPU/GPU、内存、存储和实时性要求。
- 通信：UART、CAN/CAN-FD、USB、Ethernet、EtherCAT、DDS 域和网络拓扑。
- 供电与安全：电池/电源、保险和限流、欠压阈值、物理急停、安全继电器、看门狗、制动和失联停机路径。
- 软件：ROS 发行版、C++/Python、驱动/算法包、仿真器、容器和构建系统。
- 部署：仿真、台架、实机或组合；目标主机、SSH 范围、是否允许安装、部署、重启和运动。
- 产品边界：许可证、是否公开、凭据注入方式、日志与隐私要求、交付与回滚方式。
- 验收：构建、仿真、故障注入、HIL、低速实机和任务级指标。

若任务包含视觉跟随，还要确认相机型号、输入 Topic、标定、推理设备、目标定义、丢失目标行为和隐私边界。没有相机输入时，只能把视觉跟随声明为 `blocked` 或 `partial`。

## 交付等级

| 等级 | 可声明的结果 | 最小证据 |
|---|---|---|
| L1 架构级 | 路由、接口契约、仓库骨架和待确认项 | 底盘类型、项目方向、主要技术栈 |
| L2 可构建仿真 | 可构建并可启动的仿真项目 | L1，加仿真器、几何/运动学、仿真传感器与自动化测试 |
| L3 可部署实机 | 可部署但默认禁止运动的实机项目 | L2，加真实驱动接口、编码器尺度、外参、电源、安全链和部署契约；没有测试默认值穿透 |
| L4 已验证实机 | 已在指定机器人上通过验收 | L3，加构建/部署/低速运动/故障注入/任务验收的运行证据和明确授权 |

输入验证器最多判定“具备某等级的输入条件”。L4 必须有运行证据，不能仅由 JSON 输入推断。交付时明确写出当前等级，不得把 L1/L2 产物称为完整实机项目。

## 运动学路由

| 可观察机械特征 | 分支 |
|---|---|
| 左右轮组独立驱动且无转向机构 | 差速或滑移转向 |
| 前轮有转角、车辆有最小转弯半径 | 阿克曼/自行车模型 |
| 滚子方向通常约 45 度 | 麦克纳姆轮 |
| 三个或更多全向轮 | 全向轮 |
| 每个模块可独立驱动和转向 | 舵轮/模块转向 |
| 铰接车体、摇臂或复合机构 | 特殊轮式模型 |

描述不明确时，询问哪些轮驱动、哪些轮转向、是否可横向运动、是否可原地旋转。

## 任务路由

### 从零建设

先确定目标交付等级。至少需要知道运动学、主要计算平台、执行器接口、关键传感器、目标 ROS/操作系统和仿真/实机目标。缺失硬件型号时可先定义接口和模拟器，但不要生成看似可直接驱动实机的参数。所有硬件合同值必须传播到 URDF/Xacro、控制器、固件、状态估计、仿真和启动配置；不能只更新一处参数文件。

### 故障诊断

要求现象、复现步骤、期望行为、最近变更和至少一种证据。用户只说“不能动”时按顺序检查：

1. 上游是否发布命令。
2. 命令是否经过仲裁和安全层。
3. 控制器是否 active。
4. 驱动是否接收。
5. 电机/制动是否动作。
6. 编码器与状态反馈是否闭环返回。

### 系统评审

要求目标和当前架构。若缺少指标，先建议可测指标，不要用主观架构偏好代替证据。

## 语言与平台分支

- ROS 1：只用于现存依赖，采用 catkin 隔离；明确桥接消息和单一写入者。
- ROS 2：使用 lifecycle、components、QoS、launch testing 和 rosbag2 等原生能力。
- MCU：高频电流/速度环留在 MCU；上位机发送有时限目标值，MCU 最终负责超时停机。
- Jetson：适合 GPU 感知；不要让非实时 GPU 任务阻塞底盘安全控制。
- 工控机：适合多总线和现场部署；确认实时内核与驱动支持。
- Python：不要承担硬实时闭环；适合任务编排、测试、工具和非实时节点。

## 必须暂停并询问

- 不知道底盘运动模型却要生成控制器。
- 不知道驱动拓扑、真值表、电压/电流或制动逻辑却要下发实机命令。
- 编码器尺度、轮径/轮距或传感器外参存在冲突却要部署实机。
- 物理急停、安全链或命令超时责任不明确却要解锁运动。
- SSH 目标、部署目录或权限范围不明确。
- 计划修改的文件与用户未提交改动冲突。
- 需要访问未获授权的远程仓库、云资源或凭据。
