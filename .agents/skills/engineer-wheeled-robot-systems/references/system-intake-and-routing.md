# 系统信息收集与路由

## 最小输入

只询问会改变方案的事实：

- 任务：建图、导航、巡检、搬运、跟随、竞速、室内/室外或其他。
- 底盘：轮数、驱动轮、转向轮、轮径、轮距、轴距、最大速度、载荷。
- 执行：电机类型、驱动器、编码器、制动器、急停和底层控制模式。
- 传感：雷达、相机、IMU、GNSS、轮速和安全传感器的型号、接口、频率。
- 计算：MCU、Jetson、工控机的型号、操作系统、架构、GPU 和存储。
- 软件：ROS 发行版、C++/Python、已有包、容器、仿真器和构建系统。
- 通信：UART、CAN/CAN-FD、USB、Ethernet、EtherCAT、DDS 网络拓扑。
- 约束：实时性、网络、功耗、环境、安全、许可证和交付周期。
- 权限：离线、仿真、SSH 只读、部署、重启、参数写入或实机运动。

用户给出现有仓库时，先运行 inspect_ros_workspace.py，再针对缺口提问。不要要求用户重复源码中可发现的信息。

## 默认值

在用户未指定且不会造成硬件风险时使用：

- 新项目默认 ROS 2 当前受支持的长期支持版本，并在输出中写明具体选择需要结合操作系统和供应商支持再次确认。
- 核心实时/高频路径优先 C++，编排、工具和低频业务优先 Python。
- 机器人描述使用 URDF/Xacro；ROS 2 控制接口优先 ros2_control。
- 室内二维导航优先 Nav2、robot_localization 和 SLAM Toolbox/AMCL 的可替换组合。
- 仿真优先选择与目标 ROS 发行版兼容的现代 Gazebo；GPU 传感器和 NVIDIA 加速需求再考虑 Isaac Sim；控制研究或已有模型要求时考虑 MuJoCo。
- 实机远程操作优先 OpenSSH alias 和只读诊断。

不要默认传感器精度、控制频率、轮径、轮距、坐标系方向或电机极性。

## 底盘路由

| 可观测特征 | 选择 |
|---|---|
| 左右轮组独立、不可侧移 | 差速或滑移转向 |
| 车轮有转角、满足最小转弯半径 | 阿克曼/自行车模型 |
| 麦克纳姆轮滚子约 45 度 | 麦克纳姆 |
| 三个及以上全向轮 | 全向轮 |
| 每个模块可独立驱动和转向 | 舵轮/独立转向 |
| 铰接车体、摇臂悬架或拖挂 | 特殊轮式模型 |

底盘描述不清时必须询问：哪些轮驱动、哪些轮转向、是否能侧向运动、是否允许原地转向。

## 任务路由

### 从零建设

要求至少知道底盘运动学、主要计算平台、执行器接口、基础传感器、目标 ROS/操作系统和仿真或实机目标。缺失硬件型号时可先产生接口化架构，但不得生成声称可直接驱动实机的参数。

### 故障诊断

要求现象、复现步骤、期望行为、最近变更和至少一种证据。若用户只说“不能动”，先区分：

1. 命令是否产生。
2. 命令是否通过 mux/安全层。
3. 控制器是否 active。
4. 驱动器是否接收。
5. 电机/制动器是否动作。
6. 编码器反馈是否闭环返回。

### 系统评审

要求目标和当前架构。缺少性能指标时先建立可测指标，不以主观“架构先进”作为结论。

## 技术分支选择

- ROS 1：只用于遗留依赖；隔离 catkin 工作区，明确桥接消息和单一写入者。
- ROS 2：使用 lifecycle、components、QoS、launch testing 和 rosbag2 等原生能力。
- MCU：高频电流/速度环留在 MCU；上位机发送有时限的目标值并接收带时间戳反馈。
- Jetson：适合 GPU 感知；不要让非实时 GPU 负载阻塞底盘安全控制。
- 工控机：适合多网口、现场总线和稳定部署；确认实时内核与驱动支持。
- Python：避免承担硬实时闭环；可承担任务编排、测试、工具和非实时节点。

## 立即暂停的条件

遇到以下情况立即询问用户：

- 不知道底盘运动模型却要生成控制器。
- 不知道电压、电流或驱动器接口却要下发实机命令。
- SSH 目标、工作目录或授权范围不明确。
- 命令可能让机器人运动、释放制动或覆盖固件。
- 现有用户改动与计划修改发生冲突。
- 需要创建远程仓库、公开源码或更改许可证但未获明确授权。
