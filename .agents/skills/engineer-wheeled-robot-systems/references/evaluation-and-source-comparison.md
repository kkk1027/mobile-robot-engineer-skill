# 评测与源码对照

## 目标

验证 Skill 能否根据用户输入生成安全、可执行、可验证且可泛化的方案，并识别它与真实项目的语义差异。代码重合度不是质量分：新项目应当独立实现，精确包名或代码相似甚至可能意味着泄漏、过拟合或许可证风险。

## 盲测防火墙

1. 从获准项目中只提取本轮允许的输入类别：硬件设备、技术栈、项目方向和执行边界。
2. 记录来源提交、日期和允许的目录范围；把凭据值完全排除。
3. 在生成前运行 `validate_project_intake.py`，主动提出缺口问题。用户允许测试默认值时，保留逐字段来源与适用范围。
4. 冻结补全后的输入及 SHA-256。生成阶段只能读取该输入，不能读取真实源码的包名、节点名、接口名、目录结构、参数、既有修复或历史故障。
5. 用当前 Skill 独立生成项目，输出 `solution.json`、接口契约、实现、测试和生成记录。
6. 先执行静态检查、单元测试、构建、仿真和安全闸门；不可用的运行环境必须明确记录为未验证。
7. 冻结产物文件清单及 SHA-256，之后才允许检查真实源码。
8. 对生成物和真实项目分别运行 `inspect_ros_workspace.py`，再运行 `compare_solution_to_inventory.py`。
9. 人工按系统职责、运行连线和安全行为复核，最后才决定是否修改通用 Skill。

“最近内容”必须由提交范围、日期范围或用户指定目录定义。若仓库历史过浅、只有一个巨型提交或 API 只返回部分文件，不能把该提交自动等同于“近期内容”；立即询问用户要评测的目录或文件范围。

## solution.json 最小结构

~~~json
{
  "base_type": "differential",
  "ros": ["ros2"],
  "languages": ["cpp", "python"],
  "frameworks": ["nav2", "ros2_control", "gazebo"],
  "packages": ["robot_description", "robot_bringup"],
  "package_roles": ["description", "bringup", "navigation", "simulation"],
  "interfaces": ["/cmd_vel", "/odom", "/tf", "/scan"],
  "features": ["command_timeout", "emergency_stop", "simulation"],
  "claims": {
    "navigation": {"status": "complete", "evidence": ["tests/navigation_launch_test.py"]},
    "human_following": {"status": "blocked", "evidence": ["camera input unresolved"]}
  }
}
~~~

标识符使用小写。`claims.*.status` 只能是 `complete`、`partial`、`blocked` 或 `planned`；每个 `complete` 声明必须给出可核验的文件或运行证据。包名用于追踪，不用于衡量独立生成质量；`package_roles` 才用于职责层面的静态比较。

## 100 分质量量表

| 维度 | 权重 | 核心问题 |
|---|---:|---|
| 信息理解与主动追问 | 15 | 是否区分事实、默认值、冲突和未知；是否只追问会改变方案的内容 |
| 路由与分支选择 | 10 | 运动学、ROS、计算平台、通信和仿真分支是否正确 |
| 闭环完整性 | 20 | 感知、状态估计、决策、规划、控制、执行器反馈、安全状态是否闭合 |
| 软硬件接口契约 | 15 | Topic/Action/TF/QoS/频率/单位/时钟/驱动真值表/电气边界是否明确且一致 |
| 可执行性与验证 | 15 | 能否构建、启动、测试、故障注入、回滚；证据是否支持完成声明 |
| 安全性 | 15 | 急停、仲裁、限速、命令超时、看门狗、失联停机、权限和凭据是否正确 |
| 可维护性与可观测性 | 5 | 模块边界、诊断、日志、rosbag、部署和升级回滚是否充分 |
| 泛化与独立性 | 5 | 是否从职责抽象改进，而非复制项目专用名称或代码 |

精确代码、目录、包名重合度权重为 0，只作为泄漏与许可证审计信号。静态语义覆盖率也不是最终质量分；未构建、未仿真或未实机验证的维度必须扣分并标记证据边界。

## 硬否决条件

出现任一项，本场景不得判定通过，不允许平均分掩盖：

- 选择了错误运动学或生成物要求物理不可实现的自由度。
- 实机默认值不安全，或测试默认值能直接解锁真实运动。
- 缺失命令超时、失联停机或最终停止责任；旧命令可能持续驱动。
- 绕过物理急停、安全控制器、制动或权限边界。
- 泄漏密码、令牌、私钥、网络凭据或未经允许的私有源码细节。
- 把 `partial`、`blocked`、未构建、未仿真或未实机验证的能力声称为完成。
- 存在 P0 失控路径、单点命令仲裁缺失导致多个控制源争用，且未被安全层阻断。

## 差异分类

- 语义匹配：职责、接口约束和安全行为一致，实现与命名可以不同。
- 可解释差异：实现不同但满足同一验收契约。
- 输入不可得：真实项目包含首轮允许输入中没有的信息，不能倒推为生成失败。
- 方案缺口：输入足够但生成方案没有覆盖，或声称完成却没有实现/证据。
- 方案增强：生成方案有合理的安全、测试或架构增强。
- 源码缺口：要求的安全、测试或闭环职责在真实源码中也缺失。
- 工具未知：静态扫描未识别；必须人工或运行时确认。
- 版本差异：ROS 发行版、依赖或仿真器代际不同。

不要把“文件名不同”计为方案缺口，也不要把“静态未识别”自动计为源码缺失。

## 比较输出解释

`compare_solution_to_inventory.py` 输出两类比率：

- `exact_static_match_ratio`：包含包名的字面静态匹配，只用于定位和审计。
- `semantic_static_match_ratio`：排除包名，纳入运动学、技术栈、接口、功能和包职责；仍需人工验证运行连线。

所有安全缺失都单独列出，不因比率较高而降级。最终报告需给出逐维度得分、硬否决检查、运行证据和未验证项。

## 通用改进规则

- 只有能够解释至少一个真实失败模式或一个明确覆盖缺口时，才增加通用规则。
- 项目专用名称、路径、业务文案和凭据不得写入通用 Skill。
- 改进应落在输入模式、职责、接口、安全约束、验证器或通用模板中。
- 每次改进增加匿名回归场景，防止修复一种底盘后破坏其他轮式底盘。
- 记录评测所用 Skill 提交、输入哈希、产物哈希、来源提交和工具限制。

## 用户项目取证

1. 使用只读 clone、API 快照或本地路径。
2. 不修改用户仓库，不覆盖未提交工作。
3. 先生成静态清单，再选择关键文件取证。
4. 引用仓库内相对路径和行号；报告中的凭据值必须显示为 `[REDACTED]`。
5. 制动、固件和硬件行为无法静态确认时明确标注，不以代码存在替代实机证据。
