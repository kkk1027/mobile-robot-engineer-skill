# 方案评测与源码对照

## 目标

验证 Skill 能否仅根据用户输入生成可执行方案，并识别与真实项目的差异。不要把真实源码细节泄漏给生成方案的步骤。

## 独立评测流程

1. 保存用户侧场景：任务、底盘、硬件、ROS、仿真器、约束和权限。
2. 在不读取目标源码的上下文中使用 Skill 生成方案。
3. 保存结构化 solution.json。
4. 对目标仓库运行 inspect_ros_workspace.py，生成 inventory.json。
5. 运行 compare_solution_to_inventory.py。
6. 人工复核关键文件、配置和源码行，修正静态分析误报。
7. 分类差异并决定是否修改 Skill。

## solution.json 最小结构

~~~json
{
  "base_type": "differential",
  "ros": ["ros2"],
  "languages": ["cpp", "python"],
  "frameworks": ["nav2", "ros2_control", "gazebo"],
  "packages": ["robot_description", "robot_bringup"],
  "interfaces": ["/cmd_vel", "/odom", "/tf", "/scan"],
  "features": ["command_timeout", "emergency_stop", "simulation"]
}
~~~

值使用规范化小写标识。允许增加 explanations、assumptions 和 acceptance_tests；比较脚本只读取已知字段。

## 差异分类

- 正确匹配：方案与源码职责或接口一致。
- 可接受替代：实现不同但满足同一契约。
- 方案遗漏：源码存在且场景需要，方案未覆盖。
- 方案臆测：方案声称存在但源码无证据。
- 源码缺口：方案指出的安全/工程能力在源码中缺失。
- 版本差异：依赖发行版或上游版本变化。
- 静态分析未知：需运行时、硬件或文档才能确认。

不要把“额外文件”自动视为方案错误，也不要把“未静态识别”自动视为源码缺失。

## 评分

分别评分：

- 路由准确性：底盘、ROS、计算平台、仿真器选择。
- 闭环完整性：命令、控制、执行、反馈、状态估计、规划和安全。
- 接口准确性：包、Topic、TF、QoS、参数和生命周期。
- 可执行性：步骤、命令、依赖、验证和回滚。
- 安全性：权限分级、超时、急停和停止条件。
- 证据质量：文件/行号、运行结果和不确定性。

任何 P0 安全遗漏使该场景失败，不用平均分掩盖。

## 迭代规则

- 只从至少一个真实失败或多个一致样本抽象通用规则。
- 项目专有名字放在评测数据，不写入通用 SKILL.md。
- 规则加入后重新跑所有已有场景，检查回归。
- 记录来源提交和评测日期，避免用旧代码评判新版本。

## 用户项目

对用户项目获取授权后：

1. 使用只读 clone 或本地路径。
2. 保留用户未提交改动，不修改样本。
3. 先生成静态清单，再选择关键包深入。
4. 关键结论引用绝对/仓库相对路径和行号。
5. 二进制、固件和硬件行为无法静态确认时明确标注。
